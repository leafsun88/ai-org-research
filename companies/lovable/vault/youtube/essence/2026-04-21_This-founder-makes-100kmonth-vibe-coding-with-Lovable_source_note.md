# This founder makes $100k/month vibe coding with Lovable

> 以下每个 insight block 后面只放一条较长的中文译引，不拆成碎句。

## Lovable 不只是在做工具，它已经能撑起一个月入 10 万美元的服务业务

这条访谈最直接的信号是：Lovable 不只是“帮人做 demo”，它已经可以成为一家代理机构的生产底盘。Harry 的收入规模和团队增长说明，这不是单个自由职业者的零散接单，而是一套能持续放大的交付流程。对 Lovable 来说，这种 agency use case 很重要，因为它证明工具的价值不只是自用，还能变成可复制的商业服务。

“现在我们每个月大概能做到 10 万美元收入，而且按这个速度看，到年底有机会做到 120 万美元。当然也可能继续往上跑。这大概是我们这家 agency 见过的最成功阶段了，最近几个月的增长真的很疯狂，团队也因此跟着扩起来了。说实话，这就是一个特别让人兴奋的空间。” —— Harry Roper, No Code MBA, 2026-04-21

## 他的产品化方式不是“低价开发外包”，而是 30 天快速原型

Harry 的定位很清楚：不是把 Lovable 当成便宜版外包，而是把它当成快速把产品交到客户手里的方法。企业和 startup 两类客户都在买“速度”和“可验证性”，而不是一开始就买完整定制开发。这个模式很像 AI 时代的 new agency playbook：先把东西交出来，让客户尽早反馈，再决定后续投入。

“我们现在主要有两类客户，一类是企业，一类是 startup。我们也在做一些偏 B2C 的东西，比如现在正在给迪拜一家很大的阿语学校做一个 edtech 平台；另一边，我们也在和 Dude Wipes 这种在美国很有名的品牌合作，给他们做内部工具来提速。我们还在摸索长期到底该往哪走，但核心思路一直很简单：怎么在 30 天左右快速做出一个原型，把产品先放到用户手里，让他们尽快给反馈，然后基于反馈继续往前推进。” —— Harry Roper, No Code MBA, 2026-04-21

## 这套工作流的核心，是 Lovable、Supabase 和 Cursor 的组合拳

Harry 讲 stack 的时候其实很坦白：大部分工作都在 Lovable 里完成，剩下再用 Supabase 和 Cursor 处理部署与细节修改。这里反映的不是某一个工具万能，而是一种很现实的分工方式：Lovable 负责主生产，数据库和局部改动由更专业的工具补位。对想理解 Lovable 在真实业务里怎么被用的人来说，这一段很值钱。

“就 stack 来说，我们现在就是 Lovable、Supabase，再加上一个部署工具，然后用 Cursor 做一些小的代码修改。但我会说，我们大概有 80% 到 90% 的工作都在 Lovable 里完成，代码库里 80% 到 90% 的代码也是 AI 写出来的。这个时代真的太疯狂了。” —— Harry Roper, No Code MBA, 2026-04-21

## 客户买单的不只是工具，而是信任和“最后 60%”

Harry 这段说得很像一个成熟的代理机构 owner：客户不是因为知道 Lovable 就来，而是因为他们相信你能把事情讲清楚、做完、收尾。很多 founder 甚至已经先自己在 Lovable 里搭了前端雏形，只是卡在“怎么把所有东西连起来”。这说明 agency 的价值正在从“从零做”转向“把最后一段拼好”，而内容和关系是这层信任的入口。

“我们过去一年很刻意在做的一件事，就是尽量多做内容、尽量多建立关系，这样人们看到我们知道自己在干什么以后，就会觉得‘这个人值得信任’。更有意思的是，现在有些 founder 其实已经自己在 Lovable 里做出了前端 mockup，也把想法拼得差不多了，他们能给我们看自己想要什么，但他们卡住的是：这些东西怎么连起来？UI 上应该跟什么趋势走？有哪些最佳实践？所以我们现在经常遇到的是一些很酷的 founder，他们已经做了很多前期工作，只差最后 60% 的帮助。” —— Harry Roper, No Code MBA, 2026-04-21

## 真正能交付生产级应用的人，得先理解底层技术

这段是整期里最像“生产经验”的部分。Harry 很明确地说，尤其在金融类客户里，不能只会点工具；你得知道 Supabase、RLS 这些东西怎么工作，得知道平台怎么拼起来，甚至得知道 AI 到底在干什么。Lovable 可以把开发门槛降下来，但它没有替代对技术边界的理解，反而把这种理解变得更重要。

“我们和很多金融公司合作，所以一直很注意要把底层技术弄明白。我觉得很多人会犯的一个错，就是他们不理解 Supabase 是怎么工作的，不理解 RLS 是怎么工作的，RLS 其实可以简单理解成 Supabase 的一种权限机制。哪怕你不想看代码，至少也要理解平台、理解这些东西怎么拼起来，这会极大提升你的流程。很多人用 Lovable 或别的工具时，会部署一些东西、做一些改动，但其实并不清楚 agent 在做什么；只要你对它的工作方式有一个大概了解，你在实际项目里就会顺很多。” —— Harry Roper, No Code MBA, 2026-04-21

## 生产环境不是空话，renameify 的 launch 已经跑通过了

Harry 不只是口头说 Lovable 能做生产，他给了一个实战验证：renameify.co 上 Product Hunt top 10，首日 7000 用户，系统没崩。这个例子很关键，因为它把“vibe coding 适不适合生产”从抽象争论拉回到真实用户压力下的表现。对 Lovable 的外部信誉来说，这类案例比口号有用得多。

“去年我们做了一个叫 renameify.co 的 startup，上了 Product Hunt top 10，launch 当天有 7000 个用户进来，结果什么都没坏，幸好。很多人也许还是会有点犹豫，但 Supabase 是个很强的工具，Lovable 写出来的代码也相当不错，虽然不算惊艳，但也很扎实。所以我不会劝任何人害怕它，真的只要你明白 AI 在做什么、自己在做什么就行。” —— Harry Roper, No Code MBA, 2026-04-21

## 用 Lovable 的人，最后学会的往往不只是工具，而是读代码

Harry 的背景其实挺混合：早年做过开发，后来转设计、自动化，再回到 Lovable，最后反而学会了 React 和 SQL。这个轨迹说明 Lovable 的价值不只是让非技术人“完全不懂也能做”，而是让原本有一点技术底子的人快速跨过理解门槛。更现实的一点是，哪怕不会手写代码，只要能读懂代码，你在 AI 时代就会明显更能驾驭工具。

“我大概 16 到 19 岁的时候做过开发，那时候就是玩游戏、折腾东西，后来也拿过开发工作，写过一点代码，但那从来不是我最擅长的事。所以我后来转去做网站设计，更多像设计师；再后来又做了自动化，开始给客户用这些 no-code 平台做流程。过去这一年里，因为一直在用 Lovable、接触这些东西，我发现自己学会了 React，也学会了 SQL。就算我不会从零写代码，至少我现在能读得懂大概发生了什么，知道这条路是不是对的，或者某个地方是不是犯了错。” —— Harry Roper, No Code MBA, 2026-04-21

## AI 的限制很现实，所以最好的方法是小步改、窄范围改

Harry 对 AI coding 的边界讲得很直：上下文就那么多，大项目会卡，模型有时还会随机删东西。所以真正有效的方法不是写超长需求文档，而是把改动收窄到很薄的一层，一步一步推进。这个实践经验很接地气，也很适合拿来写 Lovable 的现实使用边界。

“现在 AI 还有一个很现实的问题，就是它能看到和吐出来的上下文都有限。所以当项目变得特别大时，有时候它就会在编辑上出问题。还有一种情况，就是 AI 会莫名其妙删掉一些东西。所以我现在会强调小步走、改动要精确，尽量把每次变化控制在一条很窄的线上。很多人一上来就写一个巨大的 prompt，或者一大坨 requirements file，我反而觉得那通常不是最好的方式。现在的 AI 还不是那种我们幻想里的超级武器，它不能一条 prompt 就把 app 全做出来，你得慢慢来，一步一步推进。” —— Harry Roper, No Code MBA, 2026-04-21

## 卡住的时候，最聪明的动作往往是 reroll

这条建议很像工程师版的“别死磕”：如果 checkpoint 以前是好的，就回去重新来过。Harry 说得很明白，AI 时代的一个优势就是重来成本低，所以最没效率的事情反而是对着一个坏掉的方向硬磨。对 AI coding 现场来说，这比传统“把 bug 修完”为主的心态更适用。

“如果你真的卡住了，我很喜欢 Y Combinator 前阵子播客里提到的一个思路，叫 reroll。意思就是：如果你卡死了，就直接回到一个还正常工作的 checkpoint，然后重新来一遍。这个在传统开发里其实不太常见，因为我们总是倾向于继续写、继续修、继续往前挤；但现在只要几秒钟你就能重新起一个版本，所以直接回到工作正常的地方，再换一种方法试一次，会容易很多。别在一个死胡同里耗太久，很多人最可惜的就是在那里浪费 credits 和时间。” —— Harry Roper, No Code MBA, 2026-04-21

## 会做 design engineer 的团队，和会卖技术的创始人，最容易跑出来

Harry 的招聘和角色分工也很有意思：还是会优先招传统前端开发，但真正吃香的是既懂代码又懂设计的 design engineers。与此同时，他自己也会把更多时间放到销售和增长上，但复杂项目仍然要亲自碰。这个结构说明 Lovable 代理机构并不是“人人都会一点点”的平均主义，而是对跨界能力有更高要求。

“我们还是会尽量招传统前端开发，因为我前面也说了，知道怎么重写代码、知道怎么更手动地用 Cursor 去修 bug，是很有帮助的。但我们更重视的是 design engineers，也就是那些既懂怎么写代码、又懂一点设计的人，这样他们才能把东西做得漂亮。大多数开发者其实很快就能上手，通常一两周之后就知道怎么用这个平台了。而我自己这边，团队能帮我扛很多事，我主要想把重心放在增长和销售上，但碰到更复杂的项目，我还是会自己下场，因为有技术理解，销售的时候真的会顺很多。” —— Harry Roper, No Code MBA, 2026-04-21

## 最好的获客方式，是先给自己做一个好项目，再持续公开展示

Harry 最后给的建议很朴素，也很有效：先做出你自己的项目，再把它持续展示出来，客户自然会找上门。比起冷 DM 海量轰炸，这种方式更像真实证明。再往前走一步，他对 agency 的目标也很直白，就是把自己做成世界第一的 Lovable 机构，而且每次客户会议都要带着 presentation 和计划，这才是能把小单做成大单的动作。

“如果你现在想开始做一点点 freelancing，我觉得第一步就是先给自己做一个很酷的项目。你有了这个之后，就能像我刚才说的那样去证明：最好的获客方式是什么？就是把你做出来的东西直接展示给客户看，给他们看这玩意儿有多酷、有多强。比起冷 DM 一千个人，这个路径有效得多。我们自己的经历就是这样：我一开始在 alpha 版本里用这个工具，给自己做了很多项目，后来把它给几个老客户或者新联系上的人看了，他们立刻就看到了价值，马上愿意合作。” —— Harry Roper, No Code MBA, 2026-04-21

## 会议里带着 presentation，代理机构才会显得真正可控

代理机构的成交和续约，不只是看交付速度，还看客户能不能感受到你在控场。Harry 讲得很直接：每次会议都该有 presentation、有节奏、有结论，不然客户会觉得你只是来问问题的。这个动作看起来很朴素，但它其实是服务业务的管理中枢，尤其适合那种靠快速原型和高频反馈推进的 Lovable 代理模式。

“还有一件很多人没意识到的事：每次去见客户，一定要带 presentation，一定要有计划，因为很多人其实不会这么做。你得让客户感觉自己被照顾到了，知道项目现在在什么阶段、什么时候该给反馈。我的很多成交，其实就来自于每次会议都有一套完整的 presentation，把客户从起点带到终点，然后这次会就能体面地结束。” —— Harry Roper, No Code MBA, 2026-04-21

## 想做第一名，就得把客户范围和增长天花板看清楚

Harry 并没有把 agency 未来想成一个小而美工作室，而是直接拿“世界第一的 Lovable agency”作锚点。他看的是客户价值最大化、收入规模上限，以及未来还能往 tens of millions 的传统软件 agency 规模靠近多少。这个判断很重要，因为它说明 Lovable 不是只能做个体副业工具，它也能成为一条完整的服务业务扩张路线。

“如果说 agency 的未来，我们最想做的就是一直保持自己是全球第一的 Lovable agency。我们会继续服务客户，也会继续想清楚 Lovable 在什么客户身上最能发挥价值。对我来说，天花板其实很高，因为我见过别的传统软件 agency 做到过几十万美元甚至更多的收入。总之，愿景就是继续把这件事做大，看看到底能长到多远。” —— Harry Roper, No Code MBA, 2026-04-21
