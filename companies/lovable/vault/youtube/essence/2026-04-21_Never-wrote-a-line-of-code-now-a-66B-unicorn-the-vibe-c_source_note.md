# Lovable 不是只卖“会写代码”，而是把 non-engineer 变成产品生产者

## 不会写代码的人不是劣势，关键是敢试和懂软件逻辑

Lazar 把 Lovable 的核心能力讲成了一个很简单的判断：真正的差异不是“会不会写代码”，而是你愿不愿意带着一点无知去试，去把规则改写出来。对他来说，不懂音乐理论的人反而可能更敢把规则打破；同样，不是工程师的人也可能更快接受 vibe coding，因为他们不会被旧流程和旧范式拖住。这个判断对 Lovable 很重要，因为它解释了为什么产品的用户扩张不是先从工程师开始，而是先从愿意试、愿意迭代、愿意把软件当成表达工具的人开始。

“我觉得规则就是被人一点点改写出来的。那些不懂规则的人，往往会一边走一边自己发明规则，而且有时候他们正好会故意走到规则外面去，这反而是好事。因为说实话，我们自己也不知道这些 AI app builder 背后的模型到底能做到什么程度，连模型作者有时候都未必知道。所以我的建议很简单：别急着先把自己塞进‘我是不是工程师’这个框里，先试，先看看会发生什么。Lovable 之类的工具给了人一个很大的能力窗口，让你不用等资源、也不用等许可，就能把创造力先跑起来。” —— Lazar Jovanovic, YouTube, 2026-04-21

## 软件难在维护，所以要把前后两头都文档化

Lazar 其实没有把 AI coding 的难点讲成“生成得够不够快”，而是讲成“生成之后能不能维护”。他的解法非常典型：先写清楚 PRD，把预期、关系和依赖说透；做完以后再把 source of truth 反向补回来，让后续的 agent 知道哪里该改、哪里不能动。这个思路说明 Lovable 的价值不是一次性 demo，而是把维护软件的认知成本降下来。

“如果你要做一个以后还得维护的功能，那你不能只会往前冲。你必须把它正着和反着都记录清楚。先在 build 之前把东西写明白，通常我会用 PRD，把我们要做什么拆开讲清楚；等功能做完，不管是我做的还是工具做的，我还会回头把 source of truth 补上，说明它是怎么被构建出来的、各个关系和依赖是什么。这样一来，就算以后某个地方坏了，你也能把 agent 直接指到正确的位置去修。这个流程不一定完美，但它会让维护软件这件事容易很多。” —— Lazar Jovanovic, YouTube, 2026-04-21

## Lovable 的目标不是单一 consumer 玩具，而是覆盖个人、业务和企业内工具

Lazar 对 Lovable 的产品边界说得很明确：它既可以是给孩子做个小游戏，也可以是 solo founder 做出能赚钱的产品，还可以是企业内部快速原型和内部工具平台。这里最值得注意的是他提到的 feature adoption tool，这说明 Lovable 在他眼里不是“只做 app 的玩具”，而是能嵌进真实组织工作流的轻量产品层。这个定位对后续 GTM 很关键，因为它把用户面从“想做个项目的人”扩成了“任何需要快速验证或内部工具的人”。

“我们真正想做的是让任何人都能做任何事。你如果只是想给儿子做一个游戏，因为他在车上一直闹，那你也可以在 Lovable 里做；你如果想自己做一个能赚钱的 business，也可以；你如果是一个企业，想在内部更快地做原型、缩短从想法到生产软件的周期，Lovable 也非常适合。甚至还有一种场景是内部 tooling，给工程团队或者非工程团队用的那种真正企业级工具。我的判断一直很清楚：如果代码曾经是门槛，我们就应该把它解决掉。” —— Lazar Jovanovic, YouTube, 2026-04-21

## 真正的企业扩张不是先签合同，而是先让 internal champion 先用起来

这段访谈里最接近 GTM 现实的一点，是 Lazar 把 Lovable 的企业扩张描述成 B2C2B：先有人在公司内部做了一个好玩的工具，开始在同事之间传播，再由这个内部 champion 影响管理层和采购流程。这个路径比“我去敲门卖合同”更符合 Lovable 的产品特性，也解释了为什么内部 adoption 比 logos 更重要。

“我们现在确实在非常快地扩 go-to-market，而且收到很多 inbound。很多大公司，像 ride-sharing、telecom、graphics chips、banking 这些领域，都已经在用 Lovable 了。但我们也很清楚，单靠签合同不等于真正成功。最有效的路径往往是 B2C2B：公司里先有一个人把 Lovable 玩透了，做出一个工具，开始跟同事分享，然后这个内部 champion 再把它往上推。那个时候我们才能真正把 adoption 放大。所以我们的 obsession 不是 logo，而是内部使用率。” —— Lazar Jovanovic, YouTube, 2026-04-21

## 日活应用和变现路径，比“做出来了没”更能定义产品价值

Lazar 还给了一个很清晰的产品度量标准：Lovable 关心的不是用户是不是点过一次，而是他们是不是在持续 build、持续发功能、持续接近 monetization。这个指标非常适合用来理解 Lovable 的产品化方向，因为它把“生成应用”直接拉到了“应用是否真的有生命”的层面。

“我们内部会看一些非常用户中心的指标。比如我可以说，daily active apps 是其中一个。我们想看到人们真的在 build，而且是在持续地加那些别人真的会用的功能。还有一些指标会看他们怎么变现。我们当然不可能直接看到 Stripe 下面的钱是怎么流的，但我们会尽量让人用 Lovable 去处理变现路径里的更多环节。我们的目标其实很简单：给用户一条更容易 build 的路，也给他们一条更容易 monetize 的路，然后再让我们自己更容易看到，我们到底帮他们把成功指标推进了多少。” —— Lazar Jovanovic, YouTube, 2026-04-21
