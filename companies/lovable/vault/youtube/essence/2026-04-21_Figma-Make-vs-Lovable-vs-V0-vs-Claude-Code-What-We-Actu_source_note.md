# Lovable 在 PM 工作流里更像早期 prototype 机，而不是最终交付层

## Lovable 最适合 very early stage idea generation 和 quick prototyping

这条视频把 Lovable 的位置讲得很准：它不是用来和设计系统、生产代码抢位置的，而是用来把 PRD 迅速变成可见的东西。Diane 明确把 Lovable 放在最早期的想法生成阶段，强调它对非技术 PM 和 stakeholder 特别友好。这个定位很重要，因为它把 Lovable 从“全能 builder”拉回到“早期发现和概念验证工具”，而这正是它和 v0、Figma Make、Claude Code 的分界线。

“Lovable 是给 very early stage 用的。你有一个想法，已经写好了 PRD，想很快生成一些东西来把概念可视化，它特别适合这个阶段。我最喜欢它的一点是，它对非技术用户很友好，PM 和 stakeholder 都能很快上手，而且它的界面直观、提示也很聪明。只是我个人不会把它当成最终设计交付工具，我更愿意把它拿来做早期的 idea generation 和 quick prototyping。” —— Diane, The Design Project, 2026-04-21

## v0、Figma Make 和 Claude Code / Cursor，分别接住不同阶段的工作流

这段材料的价值在于它把工具分层讲清楚了：v0 是给已有设计系统的人，Figma Make 是给已经深度在 Figma 里工作的团队，Claude Code / Cursor 则是给真正要改前端代码的人。Lovable 放在最前面并不是因为它最好，而是因为它最快、最适合把模糊想法变成可讨论的原型。这个排序对理解竞争格局很有用，因为它说明 Lovable 的护城河不在“最终生产代码”，而在“最快把想法推到可讨论状态”。

“如果你们团队已经在用设计系统，那 v0 会更适合，因为它能直接生成 production ready 的 React 组件，代码可以直接拿去用。Figma Make 则更适合那些本来就活在 Figma 里的团队，尤其是设计师已经在那边工作的时候，几乎没有 context switching。至于 Claude Code 和 Cursor，它们就更像是让 PM 真正去改代码、修小 bug、调 spacing、调 copy 的工具，不是做快速原型那一层，而是做真实生产变更那一层。每个工具都对应一个阶段。” —— Diane, The Design Project, 2026-04-21

## 真正的 PM 工作流不是押一个工具，而是用更多原型换更好的决策

视频后半段最有用的不是某个工具的界面，而是她对 PM workflow 的判断：这些工具的核心价值是让团队在更短时间里做出更多 prototype，然后把 human refinement、brand application 和 accessibility review 留给人来做。换句话说，Lovable 的价值不是“替代设计和开发”，而是让产品探索更密、更快、成本更低。

“这些工具的真正价值，不是说你十分钟就能产出一个 prototype，而是你可以在三十分钟里做五个 prototype。更多的迭代意味着更好的产品判断。PM 现在不再被设计和开发卡住，你可以先简单开始，先用一个工具把自己的思路跑出来，再把它交给设计师和开发者去打磨。只是你不能把所有工具一次性全上，最好选一个最符合你当前阶段的，然后慢慢把自己的能力练出来。” —— Diane, The Design Project, 2026-04-21
