# Lovable / 20VC Windsurf 2025-06-02 — Source Note

> 这是一篇竞品 founder / 行业判断 source note。它不代表 Lovable 内部事实，但能作为 AI coding / app builder 竞争格局、application layer 价值归属和组织速度的高相关反方材料。以下每个 insight block 后面只放一条较长的中文译引，不拆成碎句；如需逐字核验，回到 `source_path`。

## Varun 对 startup idea 的判断，是乐观和现实必须同时存在

Varun 用 Windsurf 多次 pivot 的经验提醒一个很适合 AI 应用层的判断：非共识 idea 不自动等于好 idea，founder 既需要 irrational optimism，也要每天问公司是否还有存在理由。这个观点可以作为 Lovable 研究里的方法论提醒：AI 应用公司 early traction 很快，错误方向也会很快被漂亮指标掩盖。真正强的组织不是坚持最久，而是能在证据变化时迅速重新定位。

“创业最难的部分，是你脑子里必须同时持有两种几乎相反的信念：一边是 irrational optimism，因为如果你完全没有乐观，你就什么都不会做，甚至不会起床；另一边是 uncompromising realism，你每天都要问自己，我们还有存在的理由吗？如果答案其实没有，你就必须非常快地改变想法，并且非常快地重新定位整个公司。startup 里有一件很奇怪的事：你不会因为在同一件错事上坚持更久而赢得奖项；你告诉同事、投资人和朋友说自己还在做原来的事，也许当下感觉更好，但如果最后失败，没人会在乎你坚持了多久。” —— Varun Mohan, 20VC, 2025-06-02

## First mover 的价值不是先发布，而是组织愿意自我打断

Varun 对“time to clone 变短后 first mover 还有没有意义”的回答很适合 Lovable。先发不只是拿到用户，而是说明公司愿意快速改变、投不增量的新技术、在范式变化时打断自己；同时它能让团队更早看到哪些路走不通，并把这些失败变成内部 DNA。Lovable 从 GPT Engineer 到 Lovable 的转向也是类似逻辑：先发不保证防御，但能让团队更早学习和纠偏。

“Being first 做两件事。第一，它是你怎样经营公司的信号；通常最先进入新范式的公司，是那些愿意 disrupt themselves、愿意非常快地改变、一直投资非增量技术、也愿意打破自己的组织。第二，它让你更快学习。如果你先进入 product 和 market，你第一个月学到的所有东西，能不能帮助你继续做出下一个进入市场的产品，并且随着时间形成 compounding advantage？很多研发里犯过的错，外部公司根本看不到；但组织内部会形成一种智慧，有些想法我们甚至不会碰，因为我们已经试过，它们并不工作。” —— Varun Mohan, 20VC, 2025-06-02

## Windsurf 的 pivot 经验说明，现金只在组织能迅速换方向时才有价值

Varun 不反对早期多融资，因为它给团队更多 at-bats，但前提是公司能真的 pivot。Windsurf 前身在 GPU virtualization 上融了 Series A，后来因为现金充足，可以免费发布 Codium，并在一个周末决定转向 code AI，即使旧业务已有几百万收入。这条材料可以用来反衬 Lovable：资本本身不是护城河，只有当组织能把现金转化成快速试错和方向重置，资本才有战略价值。

“我们在 GPU virtualization 这个想法上融了 Series A，这确实给了我们信心：如果 pivot，我们还有现金去追一个新想法。我们后来能把 Codium extension 完全免费发布，一方面是因为公司内部有基础设施经验，另一方面也是因为我们知道银行里有现金，不需要担心那些细节。所以我同意，多融资会给你更多 shots；但你必须是一家能非常快 pivot 的公司。大多数公司到了 10、15、20 人时，都不愿意彻底改变自己的想法。我们决定做 Codium，是我和 cofounder 在一个周末定下来的，周一告诉公司，大家周一就开始做，尽管之前的业务已经有几百万美元收入。” —— Varun Mohan, 20VC, 2025-06-02

## 高速 AI 公司不能同时养多条指数曲线，pivot 时要 cold turkey

Varun 说 startup 通常不是因为做很多事都做得好而成功，而是因为把一件事做得极好。产品都有自己的 R value，资源应该去增长曲线最高的产品；如果两条产品线完全不相关，就不应该同时分散资源。这个判断和 Lovable 的 focus 教训高度一致：AI 应用机会太多，团队最容易被“每件事都看起来合理”拖散。

“我对 startup 的一个信念是，公司不是因为做很多事情都做得好而成功，而是因为也许只把一件事做得非常好。能有一个好想法，对大多数 startup 来说已经是奇迹。任何产品都有某种指数增长曲线，有一个 R value；startup 正确的做法，总是把注意力放在 R value 更高的产品上。如果两个产品完全分离，分散资源去做两条不同的指数增长曲线没有意义。所以当你 pivot 时，正确答案通常是放弃过去在做的东西，因为那件事对未来业务可能没有相关性。你需要告诉员工、投资人、客户你不再支持它，这当然很吓人；但一旦做出这个艰难决定，事情反而会变得很容易。” —— Varun Mohan, 20VC, 2025-06-02

## Windsurf 和 Lovable 的分叉，是 developer power tool 与 non-developer creation layer 的路线选择

Varun 对 Lovable/Bolt 的评价很直接：这些工具聚焦 non-developer，未来会被迫加入更多 tunables 和配置；Windsurf 聚焦 professional developers 和大型 codebase，长期会自然下探到非技术用户。这给 Lovable 提供了一个竞争坐标：Lovable 从 simplicity 出发，Windsurf/Cursor 从 codebase understanding 出发。双方会收敛，但短期组织能力很不同，一个要懂用户心理和默认体验，一个要懂复杂工程系统和 developer trust。

“我们很多 Windsurf 用户其实也不是开发者，他们在 vibe coding apps，而且在做很有生产力的 use cases。我们公司内部就有一个非开发者负责 partnerships，他用我们的产品做了 apps，替代了很多 sales tool spend，内部省掉了超过 50 万美元的 sales tool，比如 partner portal、quoting tool 之类。对我们来说，一个有趣方向是让这种用户更高效；但这里总有张力。如果我们让这种用户更高效，可能会牺牲让大型 codebase 上的开发者更高效的能力。Windsurf 现在专注的是让工程师强很多；让普通人也能做工程师，是底层技术带来的结果，但 focus 仍然是前者。Lovable、Bolt 这类聚焦非开发者的工具，随着时间推移会需要更多 tunables、更多配置方式；而如果我们能深度理解大型 codebase，自然就能用很少的自然语言让非技术人员构建 apps。” —— Varun Mohan, 20VC, 2025-06-02

## 非开发者做内部工具是真实需求，但它不等于复杂软件开发已经被吃掉

Varun 认可非技术员工可以用 AI 做内部工具，也举了 Windsurf 内部 partnership 负责人替代 sales tools 的例子；但他也强调，大型 codebase、生产系统、复杂基础设施仍然需要深层工程能力。这个观点给 Lovable 的上限和边界都提供了证据：非技术 creation layer 很大，但要进入 enterprise core systems，就必须解决 codebase understanding、权限、质量、审查和生产可靠性。

“这些 sales tools 过去每年要花几十万美元，因为它们非常 bespoke，只做一个具体功能，没有单家公司愿意自己做出来。现在软件构建变得很便宜，ephemeral software 或更简单的软件变得很便宜，非技术人员就可以构建这些 apps。但这和复杂软件不是一回事。大多数软件并不只是一个带后端的网站；还有复杂的数据库系统、基础设施。Google 的大部分不是 Google 的前端，背后有复杂的 ranker，也有 Spanner 这种地理一致的数据库，可能有几千万行代码。没人会在 Figma 里设计一个东西来修改 Spanner codebase，但我们希望他们用 Windsurf 去做这种事。” —— Varun Mohan, 20VC, 2025-06-02

## 新产品验证应该用小队，而不是一上来堆十个人

Varun 对内部 product building 的组织经验很具体：未被验证的想法应该用三四个人的小队，一个有判断的小 group 去证明一个 nugget of value，而不是十个人同时发表意见。这个观点可以对照 Lovable 的早期小团队：AI 应用里的 product judgment 需要低沟通成本和强 opinionated stance；一旦 v0 粗糙版都能证明惊艳，再加资源。

“当一个产品或想法还没有证明自己有 legs 时，你应该让很少的人去做。想象十个人在一个还没被证明的东西上工作，所有人都有意见，所有人都有想法，而没有人的想法是错的，因为还没有任何东西被证明。于是 alignment 很难，大家往一个方向走也很难，还会制造沟通问题。真正有效的是，一个小 handful 的人，有 opinionated stance，出去证明一个想法。如果你能拿到一个 nugget of value，再问自己什么时候停止。我们从 hard tech 带来的一个经验是：一个真正伟大的新想法，即使是 crappy version，也已经很 amazing，已经证明了某件惊人的事；等这个 crappy version 证明有 legs，再给它更多资源。” —— Varun Mohan, 20VC, 2025-06-02

## Windsurf 视角下，AI coding 工具的 moat 是 speed 和 learning curve

Varun 对 moat 的回答是 Lovable 竞争分析里很重要的反方材料。他认为早期 AI coding startup 讲传统 moat 太早，几十个工程师做出的产品迟早会被别人复现，真正能形成优势的是速度、学习哪里会死、以及把这些教训持续 compound。这个框架同样适用于 Lovable：它的防御不在“别人不能生成 app”，而在谁更快理解非技术用户、谁能把 prototype-to-production 的坑系统化、谁能把增长和产品反馈变成复利。

“我觉得 startup 有 moat 这个想法，多数时候是太早了。你可以有全世界最好的工程师，我也觉得我们有非常 amazing 的工程师，大多数工程师来自 MIT，而且我们能招到 MIT 最顶尖的那一小撮。但即使你有 50 到 100 个工程师，放进产品里的也只是几百个 engineering-years；这不是别人最终不能在这个类别里做出东西的领域。我认为我们这个类别唯一的 moat 是 speed，就是学习哪里埋着 dead bodies，学习怎样 compound 一个 advantage。即使大公司有 distribution 和资源，如果一个 startup 有好的 insight，并且能在战略上 out-execute 另一个公司，它仍然能赢。” —— Varun Mohan, 20VC, 2025-06-02

## 大公司输在 velocity，是因为没有 startup 的 existential dread

Varun 解释为什么 hyperscaler 有分发、有品牌、有资源，却不一定做出最好的 AI coding 产品：大公司员工没有“六个月不 ship 好产品就会失业”的生存压力，startup 则必须快速学习、快速发布。Lovable 的小团队高强度 culture 和 weekly cadence，正可以放在这个竞争逻辑里理解。AI 应用层不是资源越多越赢，速度和焦虑感本身是组织资产。

“如果你看很多 hyperscalers，它们在这个类别里的产品并不是 amazing products，尽管它们已经有很多 distribution。为什么？我觉得是因为把人才组织起来、在这个类别里非常快地执行，其实很难。我们现在基本上一到两周就 ship 一个 major release。说出一个能这么快学习的大公司。大公司为什么在这样热门的市场里开发速度差？我觉得是 existential dread。startup 有一种生存焦虑：如果不够快 ship、不够快学习，就活不下去。如果我在 Amazon 这样的巨头里面做这个类别的产品，那些员工会担心如果六个月内不做出好产品自己就没工作吗？如果我们 ship 出和一些 hyperscalers 一样质量的产品，一年前我们就已经出局了，但他们现在可能仍然有工作。” —— Varun Mohan, 20VC, 2025-06-02

## In-person 对高速 AI 产品是 unfair advantage，因为它压缩了 pivot 和校准成本

Varun 明确说 Windsurf 是 100% in-person 公司，并认为同一房间、同一页、五分钟内拉人进来讨论，是速度上的 unfair advantage。这个观点和 Lovable 早期强调 office、lunch、高带宽沟通互相印证：AI-native 公司并不天然等于全远程异步。模型让执行更快，方向错的代价也更快放大，因此组织需要更高频的线下校准。

“我们现在是一家 in-person company，100% in-person。今天也许可以远程建公司，但会更难一点；你必须从一开始就是一家原则性强得多的公司。如果一家公司能让工程师在同一个房间里，必要时每五分钟就聚到同一页上，这相对于跨时区团队是一种 unfair advantage。下午两三点时，公司每个人都能立刻被拉进一个房间，这会让你的灵活性快很多。早期你的第一个想法通常不准确，所以你必须能极快 pivot，并且尽快把公司所有人 marshalled 到一起。” —— Varun Mohan, 20VC, 2025-06-02

## AI coding 品牌不能让公司放慢，用户每天都在重新评估工具

Varun 对 brand 的回答很适合写 Lovable 的风险：品牌当然有用，能让新产品更快被使用，但这个类别里品牌不能给公司放慢的权利。GitHub Copilot、Devin、Windsurf 这些名字都说明用户注意力会快速迁移。对 Lovable 来说，品牌爱和 mindshare 是资产，但只有持续 ship 才能维持。

“拥有一个大的好品牌当然有帮助；当你不断 ship 新产品时，更多人会使用它，不需要每次都从零建立 user base。但回看一两年前，GitHub Copilot 是硅谷每个人都在谈的产品；再往后是 Devin launch，三到六个月里我听到的几乎只有 Devin。我们这个类别里，几乎每天都必须重新证明自己。如果三个月里我们没有给用户 ship 出 amazing 的东西，对很多用户来说我们就会变得 irrelevant。如果我们跟不上创新速度，就会变得 irrelevant。所以，只要你能继续以 breakneck pace 创新，品牌当然有帮助；但品牌不给你作为公司放慢的权利。” —— Varun Mohan, 20VC, 2025-06-02

## Enterprise switching cost 仍然存在，但速度会不断侵蚀静态锁定

Varun 讲 enterprise business 时，承认大公司深度部署、团队协作和 wall-to-wall usage 会创造 switching cost，但他也提醒，如果产品不持续变好，再强的 switching cost 也挡不住用户迁移。这个判断对 Lovable 企业化很有用：Lovable 需要 admin、security、integration 来建立企业摩擦，但更重要的是维持 product velocity。AI coding 类工具的用户已经习惯快速切换，一旦体验落后，旧 SaaS 式锁定很难成立。

“我们公司还有一些没有真正谈到的 qualities，比如拥有一个大的 enterprise business，在很多大型 Fortune 500 公司里 wall-to-wall 使用，并且能为大型团队正常工作，这对业务有内在 switching costs。当然这还不是 Salesforce 那类 switching cost，我也不觉得我们已经把它完全建出来。但这里有一个 tradeoff：如果你过于专注让产品很难被切走，同时又没有足够快地改善产品，人们仍然会切到更好的产品上，哪怕离开你的产品并不容易。我们必须足够快地在 features 和 product 上创新，建立差异化 experience，这需要很多工作。” —— Varun Mohan, 20VC, 2025-06-02

## Enterprise 不是一个抽象 logo，它会把产品拖进 IDE、语言和既有工作流细节

Varun 用 Java 和 JetBrains 举例，说明企业采纳不是“买一个编辑器”那么简单。大公司有大量 Java code、IntelliJ/JetBrains 用户，Windsurf 必须支持这些 IDE plugin，不能只支持 VS Code users。这对 Lovable 的 enterprise 路线是提醒：如果 Lovable 从 AI-native founders 进入企业，它最终也会遇到既有系统、身份权限、代码仓库、语言栈、审批流程和组织习惯，而不能只靠一个漂亮生成界面。

“超过 50% 的收入或使用来自 enterprise。很多大企业关心的第一件事，是它们有很多 Java developers，很多大公司有大量 Java code，他们使用 JetBrains 家族里的 IntelliJ。虽然我们有 Windsurf editor，但仍然支持所有 JetBrains IDE 的 plugins，并且功能和 Windsurf editor 一样。我们 fork VS Code editor，是因为 VS Code 给不了我们足够 UI flexibility 去构建新的 agentic experience；但在 JetBrains plugins 里我们能完整提供。比如 JP Morgan Chase 是我们的客户，他们公司里超过 50% 的开发者是 JetBrains users。我们进入企业时，不想告诉他们只有 40% 的用户能用我们的产品；我们想确保所有开发者都能用。” —— Varun Mohan, 20VC, 2025-06-02

## 工程师会变成一个 spectrum，PM 也要从写文档变成能直接 build 的人

Varun 对未来工程师和 PM 的判断很有价值：会有一部分人只在自然语言抽象上操作，也能在公司里构建能力；同时 production critical systems 仍然需要能下到 weeds 的工程师。PM 的角色也会变化，不能只是告诉别人做什么，而是应该自己 build 一个版本证明想法。这个观点和 Lovable 的组织影响高度一致：AI 不是取消产品和工程角色，而是把 agency 要求抬高。

“工程师这个词会变得非常 flexible。技术越来越好以后，technical-adjacent 的人也会使用 Windsurf。未来会有一些人完全在自然语言抽象层上操作，AI 工具会向他们解释 software stack 的不同组件怎样工作，所以即使他们在自然语言之上操作，也仍然能对公司有价值。但也会一直有 production 和 critical applications，比如 JP Morgan Chase 的 transaction processing system，你大概不想 vibe code 那个系统；那种每天处理上百万、上十亿交易的系统，需要有人能下到 weeds，验证它真的合理。PM 也是一样，未来对 PM 的期待不会是‘我告诉别人去做什么’，而是你也应该去 build。AI 工具会给他们这种能力；technical PM 如果懂代码、会写代码，现在会比过去更 deadly，因为你不用给他十个人团队才能证明一个想法。” —— Varun Mohan, 20VC, 2025-06-02

## Lovable 和 Figma 只碰到软件的一部分，复杂系统仍是 AI coding 平台的长期战场

Varun 在谈 Figma、Lovable、Bolt 的 convergence 时提醒，设计到代码和前端网站只是软件的一小部分。复杂数据库、基础设施、搜索排序、长代码库和生产系统，才是工程师大量时间所在。这个观点能帮助报告避免把 Lovable 上限写得太轻：Lovable 要从 app builder 走向更大平台，就必须逐步覆盖后端、数据、部署、安全、运营和复杂系统理解；否则会停留在“软件表层”的高增长入口。

“Lovable 和 Figma 做的事情，也许只是软件的一小部分。最终大多数软件不只是一个接了后端的网站；那里有复杂的数据库系统，有各种 infrastructure。Google 的大部分不是 Google 的前端，你想想 Google 背后有多少工作，有复杂的 ranker，有 Spanner 这种地理一致数据库，可能有几千万行代码。没人会在 Figma 里设计一个东西来改 Spanner codebase。我们没有一直坐在那里想其他公司在做什么，而是在想软件工程师把时间花在所有哪些地方，以及怎样让他们效率高很多。所以设计阶段确实会被很多东西 disrupt，但那只是软件开发者最终工作的冰山一角。” —— Varun Mohan, 20VC, 2025-06-02

## Async agents 的难点不是能不能跑，而是 latency、quality 和 correctability 的组合

Varun 对 asynchronous agents 的判断很细：远程 agent 从手机发起、生成 PR 很诱人，但用户等了十几分钟甚至几小时后，对质量要求会极高；如果结果只有 90% 正确，修正成本可能很高。这个观点是 Lovable agentic roadmap 的重要风险框架：AI 自动化越异步，越需要高正确率和低纠错成本；复杂任务短期仍需要本地、快速迭代的反馈循环。

“构建我们这个类别的产品时，有三件事非常重要：latency，也就是多久拿到响应；quality，也就是正确性；还有 correctability，也就是你能多快修改。很多人觉得如果技术很好，就可以忽略这些事实，但这非常重要。如果你做一个 asynchronous experience，它花 10 分钟、20 分钟、30 分钟，甚至几小时构建东西，用户对输出的期待会非常高。你的 quality 必须非常高；如果 quality 不高，它最好很容易 correct。可是如果最终只是生成一个 GitHub pull request，哪怕 20% 错了，甚至 10% 错了，用户都会失去信心，因为他们等了很久才拿到结果。修改一个 90% 正确的东西也不容易，因为你还得理解错的 10%，这也需要时间。短期看，复杂且需要快速迭代的任务仍然会在 Windsurf 里完成；异步等待 30 分钟再去 GitHub 看，然后发现不对再拉回来，feedback loop 太慢了。” —— Varun Mohan, 20VC, 2025-06-02

## “SaaS 会坍缩成 database + agent”这个叙事被 Varun 认为太早

Varun 不完全接受“apps collapse into agents”的说法。他承认公司状态可以抽象为数据库，但 Salesforce、Workday 这类复杂应用背后有大量人类 workflow 和组织惯性，今天的 agents 还没有被信任到能在任意 workflow 中大规模写入数据库。这个观点能作为 Lovable/AI builder 的反方：生成新软件会很快，但替换大型企业系统会慢得多，因为技术能力之外还有人类流程和信任问题。

“任何公司的状态当然都可以被蒸馏成一个数据库。但我不完全相信，在短期内 Salesforce、Workday 这样的复杂应用会被另一个只做数据库、然后让 agent 跑在上面的公司取代。围绕 Salesforce 已经建立了大量复杂 workflow；我们自己搭 sales 时，也是因为大家以后都知道怎么用 Salesforce，于是我们在 Salesforce 里做 custom workflows。你说得对，这些本质上都是 database transactions，但它们影响的是人类怎样和软件一起工作。更关键的是，我不认为今天的 agents 已经好到你可以让它们在没有人类监督的情况下，在任意 workflow 里自动写数据库。现在这些 agents 更多是在读 data sources，而不是大规模修改东西，因为我们还没有足够信任这些系统。” —— Varun Mohan, 20VC, 2025-06-02

## Varun 同时批评 agent hype 和低估指数进步，这给 Lovable 的产品节奏提供了边界

Varun 认为投资人常常高估今天 agents 的能力，比如 Devin 发布时大家以为 junior developer 已经不需要招了；但他又强调人们低估六个月后的指数进步。这个双重判断很适合 Lovable：产品不能假设今天的模型已经能完全托管复杂 workflow，也不能按今天模型能力保守设计。正确节奏是在当前可靠性边界内做产品，同时为下一代能力预留 interface 和 workflow。

“我觉得很多投资人相信这些系统今天比实际更强。去年 3 月 Devin launch 出来时，所有人都兴奋于软件工程师会被替代，很多投资人说再也没有理由招 junior developer；但当时我们离用 asynchronous AI agent 替代 junior developer 还很远。另一方面，人们也没有理解 exponentials 改善得有多快。六个月以后，这些模型能做的事情会和今天非常不同。现在 agents 很擅长大规模读取系统，但人们还不信任它们大规模写入系统、写入内部数据库；这可能六个月后就会改变，因为八九个月前人们还没怎么使用 agents。” —— Varun Mohan, 20VC, 2025-06-02

## 模型供应商短期很难在单一类别跑掉，application layer 仍有空间

Varun 对 foundation model 竞争的判断是：短期很难出现一个 model provider 在某个高价值类别里无人能追，因为新技术、资本和人才会快速追赶；同时模型公司会向 app layer 上探，但也可以通过更好的 API、延迟优化和 workload specialization 服务应用公司。这个判断能支撑 Lovable 的位置：底层模型会变强且竞争激烈，应用层仍然可以通过 workflow、数据、用户体验和速度获取价值。

“短期内，我觉得不太可能出现一个 model provider 在某个非常有价值的类别里直接跑掉。scale 能带来更好模型，资本少当然会束手束脚；但我们还没到某家公司会成为定义性领导者、其他人几年都碰不到的阶段。六个月后，如果一个新技术出现，被另一家公司发现，并且能用更高效方式实现，而你还在用旧技术，你就会落后。除非你相信一家公司会拿到世界上所有资本、垄断所有好想法，否则短期不太可能有人彻底跑掉。模型公司当然会上探到 app layer，也会在 API 侧为不同应用做更好支持，比如对代码输出做延迟优化，帮我们的 workload 比其他 workload 更好。” —— Varun Mohan, 20VC, 2025-06-02

## 一人十亿美元公司的怀疑，提醒 Lovable 不能把组织价值简化成“人少”

Varun 不相信 solo billion-dollar company 叙事。他的理由很朴素：如果一个人能做，两个同样聪明的人也能做，竞争会压低 margin、增加 CAC、削弱品牌垄断。这个观点可以作为 Lovable “one-person unicorn” 叙事的反方：AI 会让小团队更强，但公司价值仍来自持续投资、分发、客户关系、品牌和组织执行，不能只靠 headcount 消失。

“我不相信 solo billion-dollar company。这个说法隐含着没有别人会在乎这个机会，但我们生活在资本主义市场里，有竞争。如果一个人能做成某件事，除非你比所有人聪明和强一千倍，否则两个和你一样聪明的人大概也能做你的 idea。一旦他们也做，就会压缩你的 margin，压缩你成为 pervasive brand 的能力，也会提高你的 customer acquisition cost。一个人要在没有持续投资的情况下跑出十亿美元产品，似乎非常难。公司是什么？公司是未来现金流的贴现总和。如果 AI 好到你完全不需要人，那我会问你到底贡献了什么；如果 AI 在构建整个东西，竞争压力反而会上升，因为其他人也能用同样的 AI。” —— Varun Mohan, 20VC, 2025-06-02

## “等到快淹死再招人”并不是乱，而是先证明公司内部能做出 1% 到 10% 的结果

Varun 对 hiring timing 的观点能作为 Lovable 岗位扩张分析的补充。他不是否定 domain experts，而是说如果公司内部连 1% 或 10% 的结果都做不出来，就说明还不到招该角色的时候。Windsurf 在招 VP Sales 前，已经没有销售人员也关掉一些大型企业客户，从而理解了 enterprise sales 的 pain，也能向候选人证明公司值得加入。这个逻辑很适合分析 Lovable 后来的 enterprise sales、security、GRC、PM 等岗位：先有真实痛点，再补 owner。

“我们有一些案例说明这个 anti-pattern。比如在招 VP Sales 前，我们公司内部没有任何销售人员，却关掉了一些非常大的 enterprise customers。正因为我们内部能做到这一点，才真正感受到 enterprise sales 会是什么样的 pain，也向我们要招的人证明了我们是一家值得加入的 awesome company。domain experts 当然很棒，但如果公司内部不能做到 1% 或 10% 的效果，那可能还不是该招这个人的时候。startup 失败不是因为内部看起来像一团乱麻，startup 失败是因为没有把正确的事做好。最坏的情况，是提前六个月招一个人，让他坐在那里等三个月、六个月、十二个月；那他就会制造一些事情来做，而我其实不在乎那些事情。” —— Varun Mohan, 20VC, 2025-06-02

## 管理上最大的反直觉，是团队规模不是 output 的函数

Varun 说自己改变最大的管理认知，是 team size 不等于 output，真正重要的是 ruthless focus。一个收入规模相同但只有一件事重要的公司，比有五个 priority 的公司容易运行得多。这个判断对 Lovable 同样有解释力：AI 应用层有无数 shiny things，团队越快越容易被新机会拉走；组织设计的核心不是加人，而是让很多人对准同一个最重要问题。

“我对管理改变最大的想法，可能是团队有多大并不是 output 的函数。真正唯一重要的是 ruthless focus。你可以有很多人，但如果这些人没有无情地聚焦在一件事上，你会非常不成功。管理一家只有一件事重要的公司，比管理一家收入同样大、但有五件事重要、五个不同 priority 的公司容易得多。回头看这当然显而易见；但在我们这样的 startup 里，总有太多事情可以做，总有几个 shiny things。你打开 Twitter，就会看到 X 被解决了、Y 被解决了、Z 被解决了。真正难的是能对事情说 no，让很多人聚焦在那一件最重要的事上，并且 ruthlessly optimize 那件事；一旦你做对了，magic 就会发生。” —— Varun Mohan, 20VC, 2025-06-02

## Windsurf 的使命是把构建技术的时间减少 99%，这解释了它与 Lovable 共享的大趋势

虽然这篇材料是竞品视角，Varun 最后给出的使命和 Lovable 的底层方向高度相通：把构建技术的时间减少 99%。两家公司路线不同，Windsurf 从 developer 和 large codebase 切入，Lovable 从 non-developer 和 creation interface 切入，但它们都在押注软件生产成本发生数量级变化。报告里可以把它作为行业共识：AI application layer 的核心变量，是谁把这 99% 的时间压缩变成可持续、可靠、可分发的产品。

“我们公司的 mission 非常令人兴奋。我不会说这只是我个人，而是整个团队：我们想把构建技术所需要的时间减少 99%。如果我们能在未来这件事如何发生的过程中扮演一个有意义的角色，我会非常兴奋。” —— Varun Mohan, 20VC, 2025-06-02
