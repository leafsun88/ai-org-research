# Winner Pattern：组织维度分析引擎

一套从组织层面判断公司是否具备赢家模式的分析框架，服务于二级市场投资决策。

## 第一性原理

**组织形态应该由"你在造什么东西"倒推出来，而不是从管理学教科书正推出去。**

炸鸡的制作方式与翻转汉堡的方式以及中式炒饭的方式应该是不同的，组织也应该不同。但如果你去看大多数公司的组织结构，你会发现非常相似，这不合理。

因此本框架的分析起点不是"这家公司的组织好不好"，而是：
1. **这家公司在造什么？**（输入是什么？输出是什么？）
2. **组织必须生活在什么样的环境中？**（稳定的？还是时常变化、随时会被攻击？）
3. **因此组织应该长什么样？**（从业务本质倒推理想组织形态）
4. **实际的组织和理想组织差多远？**（七维度评分 + 适配度）

## 使用前准备

**必须先阅读知识库**：
```
view /path/to/winner-pattern-org/references/org_patterns.md
```
知识库包含从实际投资案例中提炼的组织判断原则、经典案例、以及评分标准的详细说明。

## 两大核心功能

### 功能1：知识库查阅与维护

当用户想查阅、补充、修改组织Winner Pattern的原则或案例时：
- 读取 `references/org_patterns.md`
- 帮助用户结构化新的案例或原则
- 将新内容归入正确的维度分类

### 功能2：个案组织分析

当用户提供一家公司（通过调研笔记、访谈、或公司名称）要求分析时，执行以下流程：

---

### Step 0：业务本质画像（最重要的一步）

**在打任何分之前，必须先回答四个问题：**

**Q1: 这家公司在造什么？**
- 输入是什么？（数据？硬件？客户需求？科学问题？）
- 输出是什么？（产品？平台？基础设施？研究突破？）
- 价值创造的核心环节在哪？（研发？运营？销售？生态？）

**Q2: 环境特性是什么？**
- 稳定的还是剧变的？
- 竞争格局：赢者通吃？还是多家共存？
- 反馈周期：做一个决策多久能看到结果？（广告=天，LLM训练=月，芯片=年）

**Q3: 因此组织应该长什么样？（理想组织画像）**

基于 Q1 和 Q2，推导出理想的组织形态：

| 业务特性 | 理想组织特征 |
|---------|------------|
| 需要 fundamental breakthrough（如前沿AI研究） | 小团队、super IC 主导、弱化短期考核、鼓励冒险 |
| 需要捕捉微弱信号找 next big bet（如NVIDIA） | CEO直接采样 ground truth、不看 numbers、长期视野 |
| 需要大规模高效执行（如广告/电商） | 结果导向考核、move fast、强 manager 协调 |
| 需要生态构建和开发者运营（如平台公司） | 开放架构、开发者关系、长期生态投入 |
| 需要在高度不确定中快速适应（如Shopify） | 松散耦合、高度协调、thrive on change、频繁校准 |
| 需要在红海中找缝隙+饱和攻击（如PDD） | 超级大脑决策、赛马机制、KPI驱动、逐步出清、"60分万岁" |
| 需要递归自进化（如 AI 公司用自己产品构建自己） | 内部工具 > 外部版本、PM = 核心用户、compounding engineering、实验驱动 |
| 需要快速建立 coding 产品统治力（如 OpenAI/Codex） | Product-first 结构（PM lead 全闭环）、赛马、all-in 资源倾斜 |
| 需要精密复杂的系统工程（如芯片设计） | One Architecture 统一底座、长期稳定的技术栈 |
| 需要工业化创新、在已验证方向上做得更好更快（如字节） | 强中台复用、信号扫描+赛马+快速立项、看板驱动熵减、Context not control |
| 需要让最好的创意判断涌现（如Netflix等内容公司） | Informed Captain一人决策、无奖金+Keeper Test、全员Context透明、科学度量兜底直觉 |

这个表不是穷举——分析时需要根据具体公司的业务本质自行推导理想组织形态。

**Q4: 实际组织和理想组织差多远？**

这个 gap 就是"组织-业务适配度"的来源。

---

### Step 1：信息收集

信息来源优先级：
1. 用户提供的一手调研笔记/访谈纪要（最高权重）
2. 用户 Google Drive 中的研究文档（用 google_drive_search 搜索）
3. 公开信息补充（用 web_search 搜索财报、新闻、Glassdoor、LinkedIn等）
4. **创始人公开发言**（播客、长访谈、earnings call、博客）——对 D1 判断尤其关键。CEO 的组织哲学几乎只在长播客和深度访谈中展露，新闻稿和 PR 里看不到。如果 D1 信息不足，触发 **exec-voice-intel skill** 系统性搜索创始人公开发言

如果用户只给了公司名称，主动搜索公开信息，但要标注信息来源和置信度。

---

### Step 1.5：核心机制识别（A/B 分级）

在打分之前，先识别这家公司**真正决定性的 2-3 个组织设计**。每家公司的核心组织选择通常不超过 3 个——其余的是支撑性的。

**A类（核心机制）**——同时满足以下三条：
1. **必要性**：去掉这个机制，这家公司的核心竞争力会显著不同
2. **因果链条**：这个机制和 Step 0 推导的业务本质有直接因果关系
3. **可验证**：有具体的"战役"或业务结果证明这个机制有效

**B类（支撑机制）**：有趣但不满足以上全部三条。

**案例**：
- NVIDIA A类：T5T 信号采样系统、One Architecture 十年不变、不看 numbers 的考核
- DeepSeek A类：20-30人小团队、奖金无区分度、不招大佬只招好奇心
- PDD A类：超级大脑 10-15 人决策圈、赛马+饱和攻击、60分万岁+敢为天下后
- 字节 A类：工业化创新机制（立项-赛马-review闭环）、一鸣管理看板系统（业务360+HR看板）、"始终创业"文化+活水机制
- Netflix A类：Informed Captain（所有人讨论一个人决策）、Keeper Test+无奖金+无定期绩效评估、AVS/MVS内容度量体系

分析时，A类机制重点展开（穿透到机制运转细节），B类一句话带过。七维度打分时，A类机制所在的维度是判断这家公司的关键——如果 A类机制所在维度 ≤ 2 分，基本可以判定组织有致命问题。

---

### Step 2：七维度评分

对照知识库中的七个维度逐一打分（1-5分），每个维度的评分标准见 `references/org_patterns.md`。

**七个核心维度**：

| 维度 | 核心问题 | 权重 |
|------|---------|------|
| D1: 创始人/CEO 认知与领导力 | 组织思考是否有原生性（在别处没见过的）？是否从业务本质倒推出了独特的组织形态？ | 20% |
| D2: Key Leader 配置 | 关键岗位上是否有对的人？IC/Manager 权力结构是否与业务类型匹配？ | 15% |
| D3: 考核与激励机制 | 不看公司说什么价值观，看它怎么考核。考核在驱动什么行为？ | 15% |
| D4: 信息架构与决策效率 | 信息是否透明？CEO的判断和一线观察是否在同一通道双向流动？ | 10% |
| D5: 组织熵减能力 | 组织/员工增速是否低于业务和收入增速？ | 10% |
| D6: Talent Density | 最好的人有多少？怎么定义最好的人？talent density 是否在提升？ | 15% |
| D7: Key Bet 战略取舍 | 战略上有没有极致的取舍？是否all-in在最重要的事上？ | 15% |

---

### Step 3：组织-业务适配度评分 [Fit Score]

这是基于 Step 0 推导出来的独立评分：

**实际组织形态 vs Step 0 推导出的理想组织形态，差距有多大？**

| 分数 | 描述 |
|------|------|
| 5 | 组织形态是从业务本质倒推出来的，机制在加速业务（如NVIDIA的T5T与信号捕捉业务完美匹配） |
| 4 | 整体匹配，有少量错位但不影响核心 |
| 3 | 部分匹配，组织在某些业务上有效、在某些上失效 |
| 2 | 明显不匹配，用旧地图走新路（如Meta用Focus on Impact做LLM） |
| 1 | 组织形态和业务本质严重矛盾，机制正在拖累业务 |

如果公司有多个业务线，需要分别评估适配度。

---

### Step 4：强制报错规则

**当以下情况发生时，必须标注 ⛔ 信息不足，拒绝打分**：

- D3（考核机制）：不知道考核周期、晋升标准、impact评估方式 → ⛔ 不打分
- D6（Talent Density）：不知道核心团队的具体构成、关键人才背景 → ⛔ 不打分
- D7（Key Bet）：不知道公司的战略优先级和资源分配 → ⛔ 不打分
- 任何维度：信息仅来自公开PR/官网/CEO演讲，缺乏一手或二手验证 → 标注 ⚠️ 置信度低

**绝不硬打分。宁可报错让用户去补充信息，也不要编造没有依据的分数。**

## 评分原则

### 打分纪律

- **5分**：该维度几乎完美，是行业标杆级别的组织设计
- **4分**：该维度明显优于同行，有清晰的正向信号
- **3分**：该维度中规中矩，无明显优势也无明显问题
- **2分**：该维度存在隐患，有值得担忧的信号
- **1分**：该维度严重拉胯，是明确的组织风险
- **⛔**：信息不足，拒绝打分

### 总分计算

总分 = D1×20% + D2×15% + D3×15% + D4×10% + D5×10% + D6×15% + D7×15%

如果有维度未打分，总分标注"暂估"。

但总分只是参考，更重要的是：
- 适配度是否 ≤ 2 分？（组织再好，和业务不匹配也没用——这是最重要的单一判断）
- 是否有任何维度 ≤ 2 分？（单项致命缺陷）
- D3×D7 是否冲突？（考核短期但战略说长期 = CEO在说谎或不懂组织）

## 核心分析原则

### 第一性原理：组织是业务的函数

组织形态应该由"你在造什么东西"倒推出来。问自己：输入是什么？输出是什么？组织必须生活在什么样的环境中？特性是什么？大部分情况下是稳定的吗？还是时常变化、随时会被攻击？

每个公司造的东西不同，组织看起来就应该不同。如果一家公司的组织和其他公司长得一模一样，这本身就是一个负面信号——说明CEO没有从业务本质出发思考组织。

### 考核决定行为

不看公司说什么价值观，看它怎么考核。考核什么，员工就做什么。

### 组织思考的原生性

顶级 CEO 的组织思考应该有"原生性"——即在其他公司从未见过的组织设计。
- NVIDIA: T5T + One Architecture + 不看 numbers + 60个直接汇报 = 高度原生
- Shopify: Get Shit Done 系统 + 基于愿景而非指标定目标 + 欢迎混乱 = 高度原生
- DeepSeek: 项目内奖金无区分度 + 不招功成名就的大佬 + 20-30人极小团队 = 高度原生
- 管理学教科书里有的 = 不算原生

### 必须做到的

1. **先做 Step 0 再打分**：永远从"这家公司在造什么"开始，再评估组织
2. **穿透到机制层面**：考核怎么考的、信息怎么流的、决策谁做的
3. **区分"制度设计"和"实际执行"**
4. **标注信息来源和置信度**
5. **勇于不打分**：信息不足就报错

### 禁止事项

- ❌ 不要跳过 Step 0 直接打分
- ❌ 不要写空洞的管理学教科书式分析
- ❌ 不要用"该公司重视人才"这种废话
- ❌ 不要在信息不足时硬打分——最严重的禁止事项
- ❌ 不要把"组织和别的公司很像"当成正面评价——这恰恰是问题

## 各维度详细判断指引

### D1: 创始人/CEO 认知与领导力

核心判断：是否从业务本质出发，倒推出了原生的组织形态？

加分信号：
- CEO 的组织设计是"只有这家公司才有"的，且能解释"为什么我们必须这样"
- CEO 直接采样 ground truth
- CEO 能清晰回答"为什么我们的组织和别的公司不一样"
- 组织本身是一个递归改进系统——输出变成输入（如 Anthropic 用 Claude Code 写 Claude Code）

减分信号：
- 组织形态和管理学教科书没区别
- CEO 只看报告不接触一线
- 组织长得和同行一模一样
- **只有 penalty 没有 reward**（xAI 反面教材 → 所有人苟着 → 执行力崩溃）

反面案例速查：
- xAI：有算力有人有钱但组织烂 = 烧钱。"封建社会"文化，核心团队被拆散，高流动性摧毁稳定性
- Meta/Llama 4：Zuckerberg 定了错误的 timeline、选了错误的领导层、催生了错误的文化，架构 bug 一条路走到黑

### D3: 考核与激励机制

核心判断：不看公司说什么价值观，看它怎么考核。

**什么时候看 numbers，什么时候不看——决策框架：**

判断变量一：**反馈周期**
- 反馈周期短（天/周）→ 看 numbers 有效（广告投放、电商运营）
- 反馈周期长（月/年）→ 看 numbers 是灾难（前沿研究、next big bet、AI safety）

判断变量二：**需要的行为是什么**
- 需要"在已知方向上效率最大化" → 看 numbers（PDD、Meta广告）
- 需要"发现未知方向"或"做别人没做过的突破" → 不看 numbers（NVIDIA、DeepSeek）
- 需要"在不确定中持续创新" → 用愿景校准替代 numbers（Shopify）
- 需要"消除内耗、让人做对公司重要的事" → pass/no pass 极简考核（Anthropic）

**用错的后果**：Meta 在 LLM 上用 1.5 个月 checkpoint + Focus on Impact → 员工把评测集放进训练集、选短平快的技术路线 → 考核机制驱动了走捷径行为。这不是人的问题，是机制的必然结果。

**一票否决信号**：只有 penalty 没有 reward → 所有人被 hack 成"少做少错"（xAI）→ 致命反模式

必须获取的信息（否则 ⛔ 报错）：
- 考核周期、考核指标、考核在驱动什么行为
- 考核与业务本质是否匹配（反馈周期 × 需要的行为）

案例对比：
- Meta Focus on Impact：考核看 numbers → 广告匹配、LLM不匹配，Impact考核催生政治内斗
- NVIDIA 不看 numbers：弱化短期评价 → 员工敢做长期探索 → 信号捕捉匹配
- DeepSeek 项目内奖金无区分度：一荣俱荣一损俱损 → 团队拧成一股绳 → 前沿研究匹配
- PDD KPI+赛马+长期水位拉升：明确目标+饱和攻击+激励解题人 → 执行效率匹配
- Shopify 不看数+愿景驱动：不因无法量化就否决项目 → 产品创新匹配
- Anthropic pass/no pass：极简绩效+晋升解绑周期 → 把内耗压到最低 → 递归自进化匹配
- ⛔ xAI 只有penalty没有reward：所有人被hack成"少做少错" → **致命反模式**

### D4: 信息架构与决策效率

核心判断：不是"透明 vs 不透明"的二选一，而是三种模式，选哪种取决于决策是分布式还是集中式。

**三种信息架构模式：**

| 模式 | 核心逻辑 | 适用条件 | 案例 |
|------|---------|---------|------|
| Context 透明 | 让分布式决策者有足够信息自主判断 | 决策分散，需要每个人独立判断 | NVIDIA T5T、Shopify GSD、Anthropic Demo Book |
| 精确指令替代透明 | 中心节点足够懂，把指标拆到位，底下精确执行 | 决策集中，且中心节点有真实一线经验 | PDD 超级大脑拆指标 |
| Recipe 隔离 | 保护核心方法不被人员流动泄露 | 壁垒是 know-how | Anthropic 代码组间隔离 |

**关键判断变量**：这家公司的决策是分布式的还是集中式的？

**如果是分布式**（每个人需要独立判断）→ 必须做 context 透明。NVIDIA 的每个员工都要自己判断"什么是微弱信号"然后发 T5T——没有 context 就无法做高质量判断。Shopify 的每个 PM 独立决定产品方向——没有全局 context 就各干各的。

**如果是集中式**（中心节点定方向，底下执行）→ 不需要底下拉齐 context，精确指令就够了。PDD 的联席 CEO 亲自管仓库、做招商，充分积累一线经验后能把指标拆得极其精确——"某个阶段某个指标就是最关键的，不需要底下开很多会去拉齐信息"。底下人只需要在命题作文下赛马解题。

**⛔ 集中式的前提是"中心节点真的懂"**：PDD 有效是因为老板从一线干上来，真的懂。如果中心节点不懂一线但又搞集中式不透明 = 灾难（xAI：Elon 做决策但 infra 架构都不对，底下只能苟着）。

**Recipe 隔离与以上两种模式正交**——无论 context 透明还是精确指令，都可以同时做 recipe 隔离。Anthropic 对内做 context 透明（Dario memo、Demo Book），但对技术细节做严格隔离（post-training 看不到 pre-training 代码）。

**Meta 的教训**：信息透明 ≠ 知识透明。Workplace 让信息流通了，但 Llama 团队 knowledge gap 依然严重——"很多人只知道跑 code，出了问题解决不了"。

### D5: 组织熵减能力

核心定义：**组织/员工增速 < 业务/收入增速 = 熵减中**

计算方式：获取近1-3年的员工数增长率和收入增长率的比值。

辅助判断：
- 是否有"结构性不允许熵增"的设计（如 NVIDIA 无 BU、DeepSeek 保持 20-30 人团队）
- 还是只靠周期性大裁员（如 Meta）
- Shopify 的 CEO 龙卷风模式：CEO 直接干预所有项目 = 对组织冗余的极致对抗
- PDD 的逐步出清系统：by 部门 by 月观测熵增、人效挂钩主管绩效、双月排名汰换——最系统化的熵减监测机制

### D6: Talent Density

核心判断：最好的人有多少？怎么定义最好的人？

"最好的人"的定义必须从业务本质出发（和 Step 0 挂钩）：
- 前沿AI研究（DeepSeek型）：对新 idea 有极强好奇心和敏锐度的人，不是功成名就的大佬
- 平台基础设施（NVIDIA型）：能设计行业领先架构、有深度经验的人
- 高速适应型业务（Shopify型）：能 adapt change、在不确定中 thrive 的人

DeepSeek 的招聘标准是极好的参考：
- 不招功成名就的大佬（"成名已久的人创新包袱重"）
- 关注好奇心和对新事物的敏锐度（面试问"最近几个月新提出的idea你是否关注到"）
- 需要在一线打仗的将军，不是后方坐着看的专家

### D7: Key Bet 战略取舍

核心判断：战略上有没有极致的取舍？

加分标杆：
- Anthropic 聚焦 coding
- NVIDIA 只做 Zero Billion Market
- DeepSeek "越是别人几乎没怎么做过的，我们越愿意去做"

减分信号：
- 撒胡椒面、什么都做一点
- 战略重点频繁变化
- Key bet 是跟风而非原创 insight

### 跨维度：招聘方式（D2×D6）

招聘标准必须从业务本质倒推。"最好的人"没有通用定义。

**判断方式**：
1. 这家公司定义的"最好的人"和业务本质是否匹配？（DeepSeek 招好奇心强的人做前沿研究 = 匹配；Meta 招 impact 导向的人做 LLM = 不匹配）
2. 招聘标准和考核机制是否一致？（招了追求 breakthrough 的人但用短期 checkpoint 考核 = 必出问题）
3. 招聘标准本身是否有原生性？（和同行一样看学历看大厂经验 = CEO 没从业务本质思考人才）

**独特筛选手段参考**：
- 低报酬筛选（NVIDIA 秘密小组、DeepSeek 招在读博士）：给高薪筛出"想要高薪的人"，给低薪+高自由度筛出"真的想做这件事的人"——只适用于探索型业务
- 赛马筛选（PDD）：先高薪堆人，赛马去伪存精——适用于执行型业务
- 使命筛选（Anthropic universal culture interview）：用使命认同度筛人——适用于需要高度协作和长期主义的业务
- 能力类型筛选（Shopify 招星际争霸选手）：用非传统信号识别"在不确定中做长期正确决策"的能力

### 跨维度：开会方式（D1×D4）

开会方式不是"多好"或"少好"，而是看需要解决什么信息问题。

| 模式 | 适用条件 | 案例 |
|------|---------|------|
| CEO 混层级参会 | CEO 需要直接采样 ground truth | NVIDIA "Let's Reason Together" |
| 不开会，精确指令替代 | 指标拆得够精确，开会同步是浪费 | PDD 不开例会不做PPT |
| 轻量系统替代会议 | 项目量级大，CEO 精力有限 | Shopify GSD 录2分钟视频 |
| ⛔ Status report 常态化 | 致命反模式——信息被蒸馏，有 bias 没 ground truth | Meta War Room |

**判断方式**：问"这家公司的会议在解决什么信息问题？"如果答案是"同步进度"→ 大概率低效。如果是"CEO 采样 ground truth"或"跨团队 reason together"→ 有价值。

## 阶段适配

### 成长期（B-D轮，50-500人）
- D1/D7 权重最高：创始人认知和战略聚焦决定天花板
- D6 关注：talent density 是否够高？

### Pre-IPO / 已上市
- D3 权重提升：考核机制是否还在驱动正确行为？
- D5 核心指标：员工增速 vs 收入增速
- 适配度最关键：组织是否还能匹配业务的新阶段？

## 使用提示

- 当用户说"帮我分析 XX 公司的组织"时，先问用户是否有一手调研材料
- **永远从 Step 0 开始**，先画业务本质画像，再打分
- **先做 A/B 分级再打分**——识别 2-3 个核心机制，集中精力穿透这些机制
- 如果用户上传了访谈纪要，先用 interview-transcript-formatting skill 整理后再分析
- 分析完成后，如果用户需要写成正式的组织研究报告，触发 **org-x-ray skill** 生成深度文章
- 如果用户只需要简短输出，触发 **writing-style-analyzer skill** 以Yi风格输出
- 如果用户提供新的案例或原则，主动建议更新知识库
- **信息不足时必须报错，不要硬打分**
- **每个 A 类机制必须有"战役级验证"**——用具体的竞争胜负或业务里程碑证明机制有效，不接受"文化好""团队强"这种空话
