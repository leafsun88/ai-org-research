# Analysis2 Skill

`analysis2` 是 `analysis` 的简化版，但它不是“把逐源笔记重新摘要一遍”。它的默认写法是 **证据块拼装**：先找到几条主线，再把逐源笔记里的高价值证据块像乐高一样放到正确位置，最后做一遍顺稿，让报告逻辑顺畅、语言平实、读者能读懂。

核心原则：

```text
材料是根本，主线只是摆放材料的方法。
逐源笔记是证据层，必须保留。
逐源笔记里的证据块是报告积木，优先保留。
高价值证据块不能无声丢弃。
分析脚手架是诊断工具，默认不产出。
最终报告的主要工作是排序、连接、去重、顺稿，而不是重新压缩材料。
```

## 0. 适用场景

用 `analysis2`：

- 用户明确要“直接写最终报告”。
- 已经有足够逐源笔记，不需要再做转录抽取。
- 用户觉得 analysis1 太重，担心中间文件导致信息重复加工。
- 用户希望报告像把 essence / 逐源笔记里的证据块放到正确位置，而不是大量改写。
- 目标是复杂报告、AI 组织报告、投资判断、公司研究或 founder / org analysis。

不要用 `analysis2`：

- 高相关转录还没有逐源笔记。先用 `学习/skills/essence/SKILL.md`。
- 用户明确要求事实包、证据图、主线图、论点骨架或审计文件。用 `analysis`。
- 材料质量有明显问题，需要诊断为什么报告写差。用 `analysis` 的审计流程。

## 1. 必读但不铺开

开始前只读必要文件，不把半个项目塞进上下文。

必读：

- `Agent.md`
- `progress.md`
- `core-principles-and-lessons.md`
- `project-structure.md`
- `学习/skills/essence/SKILL.md`
- `学习/style_references/anthropic_org_investment_style_notes_2026-04-21.md`

按需读：

- `学习/style_references/anthropic_org_investment_style_reference_2026-04-21.md`
- `学习/style_references/lovable_org_philosophy_framework_reference_2026-04-23.md`
- `companies/{slug}/analysis/{Company}_org_block_report_*_recovered.md`
- 用户指定的旧稿、样例或风格 reference。

不要为了“流程完整”批量读无关 skill、legacy skill、old friends 或全部 style references。只有当报告需要对应镜头时再打开。

当用户要求“组织哲学”“组织结构”“框架重组”“参考某份组织报告结构”，或你正在为 AI 组织报告选择章节框架时，必须打开 `lovable_org_philosophy_framework_reference_2026-04-23.md`。它不是事实来源的替代品，而是框架和写法 reference。

这份 reference 的可复用结构：

- 先写 **摘要**：用 2-4 段和 3-5 个支柱，把公司组织哲学说清楚。
- 再写 **目录**：让读者先看到组织报告的地图。
- 主体用 dossier 式结构，而不是单一路线长文：`组织哲学概览 -> 使命/愿景/创立论点 -> 价值观 -> 领导层与组织结构 -> 组织结构图 -> 招聘实践 -> 办公政策 -> AI 内部运营 -> 关键引述 -> 演进时间线 -> 治理/投资人 -> 已知与未公开信息`。
- 每个大节开头先给总起，再放证据块；表格、时间线、组织图、已知/未知清单可以帮助读者吸收结构性事实。
- 写法偏“组织档案 + 判断”：标题要是组织判断，段落要解释这个安排为什么重要。不要只堆公司官网信息。
- 如果用户要求保留既有报告 block 数量，只能调整 block 所在章节和连接语，不要合并或删除 block；重组完成后用 `rg -c '^### '` 校验数量。
- 用户给的 reference 里如果带大量 citation links，正式输出默认去掉链接，只保留来源名称、时间或署名；不要把 URL 铺在正文里。

## 2. 输入层

正式报告默认只读逐源笔记：

```text
companies/{slug}/vault/podcasts/essence/*_source_note.md
companies/{slug}/vault/youtube/essence/*_source_note.md
companies/{slug}/vault/substack/essence/*_source_note.md
companies/{slug}/vault/financials/essence/*_source_note.md
companies/{slug}/vault/_optional/*/essence/*_source_note.md
```

兼容旧文件名：

```text
companies/{slug}/vault/**/essence/*.md
```

但如果同一来源同时存在旧短 `_essence.md` 和新版 `_source_note.md`，优先读 `_source_note.md`；旧短 essence 只能做审计，不让它承重。

原始转录、文章或报告只在三种情况下打开：

- 逐源笔记里的引用明显可疑。
- 关键事实需要核验。
- 逐源笔记漏掉了高价值材料，必须回源补读。

如果发现高相关原始材料没有逐源笔记，先停止写报告，回到 `essence` 补逐源笔记。

## 3. 工作流：拼装式直接写最终报告

### Step 1. 识别 target 和 source set

定位公司和正式逐源笔记：

```bash
find companies/{slug}/vault -path '*/essence/*_source_note.md' -type f | sort
```

如果没有新版逐源笔记，再看：

```bash
find companies/{slug}/vault -path '*/essence/*.md' -type f | sort
```

报告前需要知道 source 数量、来源类型和明显缺口，但不要默认产出 inventory 文件。

### Step 2. 全量逐源笔记通读

正式复杂报告前必须通读相关逐源笔记。最低标准：

- 每份读元信息、标题、每个判断块的判断段和引用。
- A 层来源要读完整逐源笔记。
- B / C 层来源至少读完标题、判断段和引用后再降权，不能只用关键词搜索拼材料。
- 读的时候在当前上下文里形成短暂材料图，但不要默认落盘成主线图。
- 重点识别哪些逐源笔记证据块可以直接搬进报告；不要先急着把它们重写成自己的话。

同时做一个临时的 **积木清单**。这个清单可以只存在于当前工作过程，不默认落盘，但写作前必须有。每个逐源笔记证据块标成四种状态之一：

| 状态 | 含义 |
|---|---|
| 直接纳入 | 原样或轻微顺稿放进报告。 |
| 合并纳入 | 和别的证据块合并，但必须保留具体事实、工具、岗位、数字或原话。 |
| 顺带点到 | 只在一段里点到，因为它重要但不需要单独成块。 |
| 有理由放弃 | 明确放弃，并能说明原因：重复、低相关、证据弱、偏离报告问题。 |

没有进入报告的高价值证据块不能默默消失。最终写完后要能回答：它放在哪里了，或者为什么没放。

来源权重只作为内部判断：

- A 层：能承重。创始人、运营负责人、客户、竞品、投资人深度材料，以及带完整引用的高相关逐源笔记。
- B 层：补边界。重复访谈、第三方评测、行业讨论、引用较弱的材料。
- C 层：只暴露缺口。低相关、二手转述、无法核验材料。

最终报告只能让 A 层证据承重；B 层补反方和边界；C 层写成待核验，不写成事实。

### Step 3. 找主线，而不是先写结论

不要先套“创始人 / 招聘 / 增长 / 企业化 / 风险”目录。先从逐源笔记里抓能串成故事线的证据块：

- 反常识点：和直觉不一样，或读者大概率不知道。
- 具体动作：招聘、会议、规划、工具、岗位、流程、销售动作、产品默认值。
- 真实数字：ARR、留存、毛利、人数、用户数、客户数、时间线。
- 冲突和反方：安全、留存、指标污染、企业交接、模型层威胁、来源偏差。
- 业务传导：组织动作如何影响采用、留存、收入质量、竞争位置或估值上限。

每个候选证据块用五个问题筛：

| 维度 | 问题 |
|---|---|
| 反常识 | 这点是否真的新，还是普通综述句？ |
| 解释力 | 它能不能解释业务为什么变强或变弱？ |
| 可验证性 | 后续能不能被客户、数据、产品事故、招聘或竞品验证？ |
| 投资相关性 | 它会不会改变市场空间、收入质量、竞争或估值判断？ |
| 具体度 | 有没有具体人、流程、工具、岗位、数字或原话？ |

高分证据块必须进入最终报告。不要为了文章顺滑删掉怪异但有解释力的细节。

这些类型默认属于高价值证据块，不能无声丢弃：

- 具体工具：Linear、FigJam、Slack、内部频道、评审系统、自动化脚本。
- 具体节奏：每周计划、演示节奏、每日发布、路线图长度、会议/午饭/办公室协作。
- 招聘仪式：工作模拟、付费试用、一周试用、强筛选招聘描述、试用期、搬迁要求。
- 岗位变化：增长工程师、销售负责人、产品经理、现场工程师、部署策略、治理/风控/合规、安全、法务、财务。
- 安全和治理动作：默认不公开、发布前检查、漏洞扫描、API key/secret 处理、认证、权限和 admin。
- 真实数字：ARR、MRR、留存、毛利、用户数、付费用户数、项目数、访问量、人数、时间线。
- 反方和边界：流失、收入口径、模型层/平台威胁、企业上线边界、来源偏差。

把证据块放进 3-6 条主线。主线是读者能顺着读下去的因果链，例如：

- 产品入口变化：谁以前不能做软件，现在为什么能做。
- 组织镜像产品：公司内部怎样要求员工也成为建造者。
- 可控地快：速度释放后，平台怎样补安全、治理和默认架构。
- 增长和信任：产品更新、员工公开表达、社区和创作者怎样变成分发。
- 商业结果和反方：收入速度、使用质量、竞争、成本和待核验问题。

主线只作为内部拼装框架，不默认单独写文件。

“AI 组织三问”可以作为内部检查表，但不要默认写成报告里的固定章节。只有用户明确要求时才保留“三问”小节。

### Step 4. 像乐高一样放证据块

落笔时，报告主体优先使用逐源笔记里的证据块：

- 证据块的判断段尽量保留原文；可以轻微改顺句子、补主语、解释术语。
- 证据块的引用使用逐源笔记里的中文译引；不要拆短，也不要只摘一句导致上下文丢失。
- 证据块标题可以调整成更适合当前章节的位置，但不要把原判断改空。
- 一个大节可以有 1 段总起，解释这一节为什么在这里、读者接下来会看到什么。
- 每组相邻证据块之间可以加 1-3 句连接，说明它们的因果关系。
- 重复证据块要合并；但合并时保留更具体、信息量更大的原判断和引用。
- 报告整体应主要由逐源笔记证据块组成，作者自己的评述只做开头、连接、解释和收束。

不要把逐源笔记证据块重写成过度抽象的“作者认为”。这会丢材料。

### Step 5. 顺稿，而不是二次摘要

全部证据块放好后，再做一遍顺稿：

- 每个大节开头必须有一段总起，告诉读者本节主线。
- 每个专有名词首次出现时要解释，尤其是 ARR、PMF、GTM、AI-native、vibe coding、Minimum Lovable Product、逐源笔记里的英文概念。
- 读者默认没看过后文；当前段落不能靠后文才能理解。
- 句子要平实、清楚，少用黑话。能说“收入是否能留住”就不要说“收入质量承压”。
- 信息密度要低一点。不要把 6 个事实塞进一句话；宁可把自然段写长一点，用完整句子解释背景、事实和含义。
- 每个自然段通常只承担一个动作：交代背景、说明材料、解释含义、连接下一段。不要在同一段里同时做完全部事情。
- 中文优先，但不要为了“全中文”硬翻专有名词。公司名、产品名、岗位名、指标名、行业常用词、原文里有语气的概念要保留英文，必要时首次出现用“英文（中文解释）”。例如 ARR、MRR、PMF、GTM、API、workflow、roadmap、demo、ship、launch、builder、AI-native、founder-mode、vibe coding、Minimum Lovable Product。
- 普通词可以翻中文，专有词不要翻得别扭。判断标准很简单：翻成中文后不像行业里会说的话，或者会损失原文语气，就保留英文；一句话里英文超过三分之一时，再把普通词改回中文。
- 正文判断段不要用“某某说 / 某某认为 / 某某觉得 / 某某指出 / 某某讲得很清楚”来带材料。引用署名放在引用后面即可；正文直接写企业动作、产品机制、组织安排和业务含义。例如不要写“Anton 说 Lovable 用 FigJam 做计划”，要写“Lovable 用 FigJam 做每周计划和问题排序”。
- 删除工作痕迹：不要说“本节将逐源笔记放在这里”“这些证据块显示”“上一版漏了这里”“这篇报告要做什么”“本轮重新看了材料”。正式报告的每一句都应该服务公司判断，而不是解释写作过程。
- 删除重复评述。材料自己已经说清楚的地方，不要再用一段空评。
- 保留反方和待核验问题，但写成人能判断的风险，不写成流程清单。

### Step 5.5. 反向查漏

报告初稿写完后，必须反向查漏：

1. 回看积木清单，检查所有“直接纳入”和“合并纳入”的证据块是否真的出现在报告里。
2. 搜索报告正文，确认高价值关键词没有丢。Lovable 这类公司至少要查：`Linear`、`FigJam`、`工作模拟`、`每周计划`、`演示`、`办公室`、`午饭`、`增长工程师`、`安全`、`流失`、`毛利`。按公司材料替换关键词。
3. 找出被抽象词吞掉的细节。例如报告只写“招聘强筛选”，但没写工作模拟、一周试用、硬核招聘描述，就是失败。
4. 如果某个高价值证据块没进报告，要么补进去，要么在待核验/边界里说明为什么没放。
5. 再做中文顺稿，清掉不必要英文和过密段落。

### Step 6. 直接写最终报告

输出路径：

```text
companies/{slug}/analysis/{Company}_{report_type}_{YYYY-MM-DD}.md
```

推荐报告结构：

```markdown
---
company: {Company}
type: final_report
date: YYYY-MM-DD
status: draft
source_layer: source_note
---

# {Company} 报告标题

## 核心判断
## 一、{主线一}
## 二、{主线二}
## 三、{主线三}
## 四、{主线四}
## 待核验问题
```

不要机械套这个目录。章节顺序要服务因果链，常见顺序是：

```text
业务本质变化 -> 组织机制 -> 具体动作 -> 业务结果 -> 反方与待验证
```

或：

```text
用户工作流如何改变 -> 公司内部如何镜像这种变化 -> 规模化后如何保持可控 -> 收入质量和竞争反方
```

默认写法是“主线 + 已摆放好的逐源笔记证据块”：

- 开头用 2-4 段讲清楚总论点。
- 每个大节开头有总起。
- 每个证据块有一个判断型小标题，下面放逐源笔记原判断段的轻度顺稿版。
- 证据块内贴近放较长中文译引，不把引用统一堆到文末。
- 每个大节末尾可有 1 小段收束，但不要再摘要所有材料。

Block 标号按章节重置：

```text
第一章：A01、A02、A03
第二章：B01、B02、B03
第三章：C01、C02、C03
```

## 4. 写作风格

写成研究员文章，不写成工作日志。

必须做到：

- 说人话。读者默认不了解公司、没读过材料。
- 标题直接给判断，不写“组织文化”“产品策略”这种空标签。
- 每段只推进一个判断，先交代事实，再解释机制，最后落业务含义。
- 重要专有名词首次出现要解释。
- 中文优先；英文只保留产品名、岗位名、指标名、原文术语或更精确的概念。
- 数字、关键事实和直接引用都能回到逐源笔记。
- 风险写成机制，不写“存在不确定性”。
- 竞争判断要直接：谁强、谁弱、强在哪里、弱在哪里。

避免：

- “这段材料说明了……”这种摘要腔。
- “某某说 / 某某认为 / 某某觉得 / 某某指出”这种访谈转述腔。
- “上一版 / 这次 / 本轮 / 本文使用 / 报告质量 / 材料丢失 / 重新阅读”这种过程说明。正式报告只写公司、业务、组织、产品、风险和证据边界。
- “AI 赋能、组织升级、深度协同、范式跃迁”这类空词。
- 用职能目录代替投资判断。
- 忠实总结每条逐源笔记，却没有二阶判断。
- 把来源权重、主线图、工作过程、修改记录写进正式报告正文。
- 为了顺滑把反常识细节、奇怪数字、具体工具和真实流程删掉。

## 5. 引用和证据规则

最终报告可以直接使用逐源笔记里的中文译引。除非用户要求英文原文，不必重新贴英文。

引用要覆盖：

- 定义问题。
- 机制如何运行。
- 具体数字、时间、比例、组织变化。
- 竞争、客户或反方判断。
- 能体现说话人语气和上下文的原话。

署名格式：

```markdown
“……” —— Anton Osika, Lenny's Podcast, 2025-03-09
```

如果逐源笔记里的引用太短、不足以支撑判断，回到逐源笔记或原始材料核验；仍不够就删掉该判断或写成待核验。

不要把分析者自己的转述放进引号里。引号里只能是来源本人表达的意思。

中文译引要尽量保留说话人的第一人称。不要把原话改成“某某在访谈里说 / 某某回应 / 某某接着说”的第三人称旁白。多人对话可以用一条较长译引承接，但引号里仍要尽量呈现“我 / 我们 / 你”的发言现场；必要的上下文放在引号外正文里，不塞进引号。

## 6. 证据限制要写，但不要写成过程日志

正式报告可以写来源偏差，但要写成投资判断的一部分：

```markdown
公开材料主要来自创始人和增长负责人，因此客户侧、员工侧、安全团队和企业采购方仍是后续验证重点。
```

不要写：

```markdown
我这次读了 24 份逐源笔记。
本轮重新过了材料。
这里按 analysis2 流程处理。
```

## 7. 什么时候才启用脚手架

默认不产出中间文件。但出现以下情况，可以临时启用 `analysis` 的脚手架：

- 逐源笔记超过当前上下文可承受范围。
- 报告主线明显发散。
- 用户要求先看结构。
- 需要审计为什么上版报告写差。
- 需要给团队留下可复查的证据图、主线图或论点骨架。

即便启用脚手架，也不要让脚手架替代最终报告。脚手架只是调试工具。

## 8. 停止条件

遇到以下情况先停，不要硬写：

- 高相关来源没有逐源笔记。
- 核心数字没有来源。
- 主判断只能来自旧报告、记忆或风格样本，不能回到正式逐源笔记。
- 需要引用但逐源笔记里没有足够上下文。
- 材料主要来自创始人/公司高管，且报告却要做强客户侧或员工侧结论。

## 9. 完成检查

最终交付前检查：

- `analysis/` 下有最终报告文件。
- 报告没有默认生成主线图、论点骨架、证据图等中间文件，除非用户要求。
- 报告直接基于逐源笔记。
- 不默认保留“AI 组织三问”固定章节，除非用户明确要求。
- A 类机制能解释业务本质，不只是职能分类。
- 高反常识、高解释力的尖锐材料没被顺手删掉。
- 关键数字和引用能回到逐源笔记。
- 正文判断段直接讲公司怎么做，不能依赖“某某说 / 某某认为”来推进。
- 正文没有“上一版 / 这次 / 本轮 / 本文使用 / 材料丢失”等写作过程痕迹。
- 引用使用第一人称中文译引，没有把原话翻成第三人称访谈摘要。
- 有反方和待核验问题。
- 正文没有研究过程、第一人称工作日志或 AI 腔。
